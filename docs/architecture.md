# Chapterize — Architecture

## Folder Structure

```
chapterize/
├── index.js          # Main entry point — all extension logic
├── style.css         # Modal and button styles
```

## Modal Flow — Two Steps

```
┌─────────────────────────────────────────────────────┐
│  MODAL STEP 1                                       │
│                                                     │
│  "Review narrative changes"                         │
│                                                     │
│  ┌─────────────────────────────────────────────┐   │
│  │ Change List (editable textarea)             │   │
│  │ — bullet points of what changed             │   │
│  │ — add / remove / reword freely              │   │
│  └─────────────────────────────────────────────┘   │
│                                                     │
│  [Next →]  [Cancel]                                 │
└──────────────────────┬──────────────────────────────┘
                       │ Next click triggers
                       │ card + situation LLM calls
                       ▼
┌─────────────────────────────────────────────────────┐
│  MODAL STEP 2                                       │
│                                                     │
│  "Review updated character and situation"           │
│                                                     │
│  ┌─────────────────────────────────────────────┐   │
│  │ Character Card Prose (editable textarea)    │   │
│  │ ⚠ AI flagged issues  (if selfcheck failed)  │   │
│  └─────────────────────────────────────────────┘   │
│                                                     │
│  ┌─────────────────────────────────────────────┐   │
│  │ Situation Summary (editable textarea)       │   │
│  │ ⚠ AI flagged issues  (if selfcheck failed)  │   │
│  └─────────────────────────────────────────────┘   │
│                                                     │
│  Turns to carry over: [4]                           │
│                                                     │
│  [← Back]  [Regenerate]  [Confirm]  [Cancel]        │
└─────────────────────────────────────────────────────┘
```

**Back** returns to Step 1 with the change list intact — no re-generation.
**Regenerate** re-runs card and situation calls using the current Step 1 content.

## ST APIs — Fully Researched

| What | How | Notes |
|---|---|---|
| Read chat | `SillyTavern.getContext().chat` | Real message objects only. No metadata header — that is `chat_metadata` (module-level variable) |
| Read character | `SillyTavern.getContext().characters[characterId]` | Full object |
| Generate text | `await generateRaw({ prompt })` | Async. Uses current API/model. Takes named options object, not a plain string |
| Save character | `POST /api/characters/edit` multipart/form-data | **Fully destructive** — must round-trip all fields. Pass `json_data` as full char JSON string, only patch `description` |
| List chats | `POST /api/chats/search` `{ avatar_url }` | Returns array sorted by `file_name` |
| Save new chat | `POST /api/chats/save` JSON body | See body shape below. `force: true` skips integrity check |
| Open chat | `openCharacterChat(fileName)` — `script.js:7522` | Async. Calls `createOrEditCharacter()` internally — must run **after** character save completes |
| Persist settings | `extension_settings['chapterize']` | Standard ST pattern. Also used for changelog storage |
| Request headers | `getRequestHeaders()` | Required on all fetch calls — adds CSRF token |

### POST /api/chats/save — body shape
```javascript
{
    ch_name: characters[this_chid].name,
    file_name: chapterName,           // no .jsonl suffix
    avatar_url: characters[this_chid].avatar,
    chat: [ metadataHeader, ...lastNMessages ],
    force: true
}
```

### Metadata header — build from chat_metadata
```javascript
// chat_metadata is a module-level ST variable (not in context.chat[]).
// Confirm whether SillyTavern.getContext() exposes it before using directly.
// If not, use the fresh-object fallback in ST_API_MAP.md §7.
const header = {
    chat_metadata: structuredClone(chat_metadata),
    user_name: 'unused',
    character_name: 'unused',
};
header.chat_metadata.lastInContextMessageId = 0;
header.chat_metadata.integrity = crypto.randomUUID();
header.chat_metadata.tainted = false;
```

### Chapter naming — full pattern
```javascript
const res = await fetch('/api/chats/search', {
    method: 'POST',
    headers: getRequestHeaders(),
    body: JSON.stringify({ avatar_url: characters[this_chid].avatar }),
});
const chats = await res.json();
const nums = chats
    .map(c => c.file_name?.match(/^ch(\d+)$/)?.[1])
    .filter(Boolean)
    .map(Number);
const nextN = nums.length ? Math.max(...nums) + 1 : 1;
const chapterName = `ch${nextN}`;
```

### Character save — use json_data to avoid field enumeration
```javascript
const char = structuredClone(context.characters[this_chid]);
char.description = newDescription;
const formData = new FormData();
formData.append('json_data', JSON.stringify(char));
formData.append('avatar_url', char.avatar);
formData.append('ch_name', char.name);
await fetch('/api/characters/edit', {
    method: 'POST',
    headers: getRequestHeaders(),  // DO NOT set Content-Type manually on FormData
    body: formData,
});
```

### Changelog storage shape
```javascript
extension_settings['chapterize'].changelog = [
    {
        date: new Date().toISOString(),
        chapterName: 'ch2',
        changeList: '- Vera revealed she knows about the dispatch\n- ...',
    },
    // ...one entry per confirmed chapterize

```

## Sequencing Constraints

```
saveCharacter()  ─────┐
                       ├── Promise.all() ──► saveNewChat() ──► openCharacterChat()
deriveChapterName() ───┘
```

**Critical:** `openCharacterChat()` calls `createOrEditCharacter()` internally at the end of its sequence. If character save has not completed first, this will overwrite the updated description with stale form state. `Promise.all` on the first two steps guarantees both are done before proceeding.

---

## Key Constraints

- Extension JS runs in the **browser** — no direct filesystem access
- All disk writes go via **ST's Node.js server** through fetch/POST calls
- `context.chat` in memory contains only real messages — the metadata header is in `chat_metadata` (module-level variable), **not** at `chat[0]`
- `context.chat` is the live in-memory array — use it directly, no file parsing needed
- `/api/characters/edit` is **fully destructive** — missing fields become empty strings — always use `json_data`
- `generateRaw` is **async** — UI must show a loading state during all LLM calls
- Step 2 LLM calls only fire on Next click — not on modal open
- `getRequestHeaders()` must be included on all fetch calls — do NOT manually set Content-Type on FormData posts
